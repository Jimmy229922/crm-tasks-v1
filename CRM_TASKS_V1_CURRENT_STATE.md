# الحالة الحالية - CRM Tasks V1

## 1) المسارات والصفحات الحالية
- المسار الأساسي للتطبيق: `/crm/tasks-v1`.
- الصفحات الحالية داخل النطاق:
  - `/crm/tasks-v1/home` -> `HomePage`.
  - `/crm/tasks-v1/risk-management` -> `CrmTasksPage`.
  - `/crm/tasks-v1/profile/:taskId` -> `TaskProfilePage`.
  - `/crm/tasks-v1/task/:taskId` -> `TaskDetailsPage`.
- يوجد تحويل افتراضي داخل النطاق من المسار الأساسي إلى `home`.

## 2) تدفق الاستخدام الحالي
- المستخدم يدخل إلى صفحة المهام `risk-management`.
- من جدول المهام يمكنه:
  - فتح بروفايل العميل عبر مسار `profile/:taskId`.
  - فتح تفاصيل المهمة عبر مسار `task/:taskId`.
- من `TaskProfilePage` يمكن فتح `TaskDetailsPage` لنفس المهمة.
- من `TaskDetailsPage` يمكن الرجوع أو الانتقال لبروفايل العميل.

## 3) وضع البحث الحالي
- البحث يتم داخل `CrmTasksPage` عبر `searchTerm`.
- منطق الفلترة الحالي يتضمن نصًا مركبًا يحتوي على:
  - اسم العميل.
  - إيميل العميل.
  - اسم الـAssignee.
  - دور الـAssignee.
  - نوع المهمة.
- أي أن البحث الحالي ليس محصورًا فقط باسم العميل أو إيميله.

## 4) وضع المودال الحالي
- يوجد مودال لإضافة مهمة جديدة فقط داخل `CrmTasksPage`.
- لا يوجد مودال لعرض تفاصيل المهمة.
- عرض تفاصيل المهمة حاليًا يتم عبر صفحة مستقلة (`TaskDetailsPage`).

## 5) مصدر البيانات الحالي
- البيانات المعروضة حاليًا تعتمد على `mockTasks` + `localStorage` محليًا.
- تحميل القائمة يتم عبر `fetchCrmTasksLocal`.
- جلب المهمة الواحدة يتم عبر `getTaskById(taskId)`.
- تجميع مهام العميل في بروفايل العميل يتم عبر `getTasksByClientEmail(email)` بعد استخراج الإيميل من المهمة.

## 6) ملاحظة بنيوية مهمة
- `TaskProfilePage` يستقبل `taskId` وليس `clientId`.
- لا يوجد Route عميل مستقل بشكل `clientId` في الحالة الحالية.
- توجد ثلاث صفحات تشغيلية مرتبطة بالتدفق: صفحة القائمة، صفحة البروفايل، صفحة تفاصيل المهمة.
